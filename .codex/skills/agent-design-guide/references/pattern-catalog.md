# Agent 模式总览

本目录基于《Agentic Design Patterns》全书中文翻译项目整理，覆盖 21 个核心章节、7 个附录，以及 FAQ / Glossary / Conclusion 的关键思想。

## 一、核心骨架模式

| 模式 | 何时使用 | 主要收益 | 主要代价/风险 |
| --- | --- | --- | --- |
| 提示词链 | 多步顺序处理 | 降低复杂度，提升稳定性 | 链路长时延高，错误会级联 |
| 路由 | 多路径分流 | 动态选择最优处理路径 | 路由错误会误分配 |
| 并行化 | 子任务彼此独立 | 降低总时延 | 并发复杂度、观测和调试成本增加 |
| 规划 | 目标复杂、需先拆解 | 从“反应式”变“策略式” | 计划可能过度或偏离实际 |

## 二、能力扩展模式

| 模式 | 何时使用 | 主要收益 | 主要代价/风险 |
| --- | --- | --- | --- |
| 工具使用 | 需要外部动作或实时数据 | 打通 API、DB、执行器 | 工具权限和错误处理复杂 |
| MCP | 需要标准化发现/接入外部能力 | 互操作、可发现、可复用 | 协议实现、安全与治理成本更高 |
| RAG | 需要私有知识或最新知识 | 提升事实性与领域适配 | 检索质量决定上限 |
| 记忆管理 | 需要跨轮上下文、个性化、长期状态 | 连续性、个性化、任务追踪 | 记忆污染、过期、隐私问题 |

## 三、质量与自治模式

| 模式 | 何时使用 | 主要收益 | 主要代价/风险 |
| --- | --- | --- | --- |
| 反思 | 输出需二次校验/优化 | 自我修正，提升质量 | 增加成本和时延 |
| 目标设定与监控 | 需要持续朝目标推进 | 让 Agent 可跟踪进度和偏差 | 指标定义不清会失真 |
| 异常处理与恢复 | 工具/网络/数据/环境可能失败 | 提升鲁棒性与用户体验 | 回退路径设计复杂 |
| 优先级排序 | 多任务、多目标、资源有限 | 聚焦关键任务，动态调度 | 优先级策略不当会饿死低优任务 |
| 资源感知优化 | 成本/时延/算力受限 | 动态选模、降本增效 | 质量与成本间存在权衡 |

## 四、组织与协作模式

| 模式 | 何时使用 | 主要收益 | 主要代价/风险 |
| --- | --- | --- | --- |
| 多 Agent 协作 | 需要专业分工或复杂协同 | 专长分离、并发、可扩展 | 协调成本、上下文同步困难 |
| A2A | 多 Agent 之间需要标准通信 | 降低耦合，提升互通 | 协议和状态同步复杂 |
| Human-in-the-Loop | 高风险、低置信度、需责任闭环 | 安全、可控、合规 | 吞吐降低、需要人工资源 |

## 五、高级智能模式

| 模式 | 何时使用 | 主要收益 | 主要代价/风险 |
| --- | --- | --- | --- |
| 学习与适应 | 环境持续变化、需长期优化 | 持续改进、个性化、自我增强 | 数据闭环与验证成本高 |
| 推理技术 | 任务需要多步推理/搜索/审议 | 提升复杂问题求解能力 | 更慢、更贵，可能过拟合推理轨迹 |
| 探索与发现 | 开放式问题、未知解空间 | 生成新假设、新策略、新知识 | 难评估，失控风险更高 |

## 六、治理模式

| 模式 | 何时使用 | 主要收益 | 主要代价/风险 |
| --- | --- | --- | --- |
| Guardrails/安全模式 | 几乎所有生产级 Agent | 控制输入输出与工具边界 | 需要持续更新策略 |
| 评估与监控 | 所有上线系统 | 可观测、可迭代、可审计 | 指标体系建设成本高 |

## 七、附录提炼

### 高级提示技术

- 先保证清晰、具体、简洁
- 优先正向指令
- 用结构化输出减少级联错误
- 用 CoT / ToT / ReAct 增强推理与行动

### GUI / 真实世界交互

- Agent 正从 API 驱动走向 GUI 与现实环境交互
- 多模态感知将成为下一阶段 Agent 的常态能力

### 框架选型

- **LangChain**：线性链与工具生态
- **LangGraph**：有状态、循环、复杂编排
- **ADK**：多 Agent、工具、企业化集成
- **CrewAI**：角色化团队协作

### CLI / Coding Agents

- 编码 Agent 最有效的形态通常是“人类编排 + 专业 Agent 团队”
- 上下文质量决定编码 Agent 上限
- 代码生成只是开始，测试、文档、审查、优化同样应该 Agent 化

## 八、组合建议

### 推荐起步组合

1. **单 Agent + 提示词链 + 工具使用 + 基础 Guardrails + 监控**
2. 如果任务变复杂，再逐步增加：规划 → 路由 → RAG/记忆 → 反思
3. 真正遇到角色分工、上下文隔离、并行收益明确时，再上多 Agent

### 常见组合

- **客服 Agent**：路由 + RAG + 工具 + HITL + 评估
- **研究 Agent**：规划 + RAG + 推理技术 + 反思 + 评估
- **企业流程 Agent**：规划 + 工具/MCP + 目标监控 + 异常恢复 + 优先级排序
- **编码 Agent**：规划 + 工具使用 + 反思 + 评估 + 多 Agent（可选）

## 九、反模式提醒

- 不要为了“看起来高级”而默认多 Agent
- 不要把 RAG 当成记忆管理的替代品
- 不要把 MCP 当成所有工具调用的默认方案
- 不要没有评估就谈“Agent 已经可上线”
- 不要没有 Guardrails 就给写操作权限
