# Agent Design Guide

用于把一个模糊的 Agent 想法收敛成可实现的系统设计。

## 何时使用

当用户要你：
- 设计一个 AI Agent / 多 Agent 系统
- 选择合适的 Agent 模式或技术栈
- 判断是否需要规划、路由、工具调用、MCP、RAG、记忆、多 Agent、反思、安全护栏、评估监控等
- 把业务流程改造成 Agent 工作流

如果只是实现某个已明确的单点功能，不需要完整模式设计时，可只参考相关章节，不必走完整流程。

## 设计流程

1. **先定义任务形态**：目标、输入、输出、成功标准、成本/时延/SLA、是否高风险。
2. **先选核心执行骨架**：
   - 单路径多步任务 → 提示词链
   - 多种请求分流 → 路由
   - 独立子任务可并发 → 并行化
   - 复杂任务需先拆解 → 规划
3. **再补能力层**：
   - 需要外部动作/实时数据 → 工具使用
   - 需要标准化外部能力生态 → MCP
   - 需要私有知识 → RAG
   - 需要跨轮状态/个性化 → 记忆管理
   - 需要专业分工 → 多 Agent 协作 / A2A
4. **再补质量层**：
   - 输出需要自检打磨 → 反思
   - 任务动态变化 → 目标设定与监控 + 优先级排序
   - 环境不稳定 → 异常处理与恢复
   - 需要持续优化成本/时延 → 资源感知优化
5. **最后补治理层**：
   - 高风险场景必须加 Guardrails/安全模式
   - 上线前后必须加评估与监控
   - 高风险决策加入 Human-in-the-Loop
   - 长周期系统再考虑学习与适应、探索与发现

## 最小设计产物

输出设计时，至少给出：
- **目标**：Agent 要完成什么，不做什么
- **主循环**：感知 → 推理/规划 → 行动 → 观察 → 修正
- **模式组合**：为什么选这些模式，不选哪些
- **状态与记忆**：上下文、长期记忆、任务状态怎么存
- **工具边界**：哪些工具可调用，权限和失败策略是什么
- **安全与监控**：输入/输出护栏、人工接管、日志、评估指标
- **资源约束**：模型分层、预算、延迟、降级方案

## 模式选择口诀

- **先骨架，后能力；先可用，后自治；先安全，后放权。**
- 不要一开始就上多 Agent；单 Agent + 规划/路由/工具往往更便宜、更稳。
- 只有在**专业分工明确、上下文过大、并发收益明显**时再拆多 Agent。
- 只有在**跨系统复用、标准化接入、动态发现能力**很重要时再优先 MCP；固定少量工具时，普通函数调用通常更简单。
